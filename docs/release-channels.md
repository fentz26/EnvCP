# Release Channels

This document describes the EnvCP release channel strategy, label system, and how PRs flow through the pipeline.

## Overview

EnvCP uses a 3-channel release strategy:

| Channel | Branch | npm tag | Docker tag | Purpose |
|---------|--------|---------|------------|---------|
| **Latest** | `main` | `latest` | `latest` | Stable, production-ready releases |
| **Experimental** | `experimental` | `experimental` | `experimental` | Pre-release features for testing |
| **Canary** | `canary` | `canary` | `canary` | Nightly builds, bleeding edge |

## Install

```bash
npm install @fentz26/envcp                  # latest (stable)
npm install @fentz26/envcp@experimental      # experimental (pre-release)
npm install @fentz26/envcp@canary            # canary (nightly)
```

Docker:

```bash
docker pull fentz26/envcp:latest
docker pull fentz26/envcp:experimental
docker pull fentz26/envcp:canary
```

## Label System

Every PR MUST have exactly one `release:*` label. The CI pipeline will reject PRs without one.

### Release Channel Labels (required)

| Label | Color | Description |
|-------|-------|-------------|
| `release:latest` | Green | Publish to stable/latest channel |
| `release:experimental` | Purple | Publish to experimental channel |
| `release:canary` | Yellow | Publish to canary/nightly channel |
| `release:skip` | Red | Do not publish (internal/CI/docs only) |

### Status Labels

| Label | Color | Description |
|-------|-------|-------------|
| `Holding` | Yellow | Work in progress. **Blocks merge** until removed |
| `ready-for-review` | Green | Ready for review, all tests passing |
| `closed` | Gray | Applied automatically to all closed PRs and issues |

### Category Labels

| Label | Description |
|-------|-------------|
| `feature` | New feature or capability |
| `enhancement` | Improvement to existing feature |
| `bug` | Bug fix |
| `security` | Security-related change |
| `hotfix` | Urgent fix requiring fast-track release |
| `breaking-change` | Breaking API change (requires major version bump) |
| `dependencies` | Dependency updates |
| `documentation` | Documentation changes |
| `testing` | Test coverage or test infrastructure |
| `configuration` | Configuration management |
| `integrity` | Data/config integrity (HMAC, checksums) |
| `audit` | Audit trail, compliance |
| `logging` | Application and security logging |
| `access-control` | Authentication, authorization, API keys |
| `notification` | Notification system (email, webhook) |
| `sandbox` | Sandbox and E2E testing |
| `codex` | Generated by Codex/AI agent |
| `discussion` | Design discussion, RFCs |

### Priority Labels

| Label | Description |
|-------|-------------|
| `priority:critical` | Must fix immediately (security, data loss, production down) |
| `priority:high` | Important, target for next release |
| `priority:medium` | Normal priority, standard backlog |
| `priority:low` | Nice to have, no urgency |

### Lifecycle Labels

| Label | Description |
|-------|-------------|
| `stale` | No activity for 60 days, auto-closes in 7 days |

## How It Works

### CI Label Check

When a PR is opened, updated, or has labels changed, the `label-check.yml` workflow runs:

1. **Check for `Holding` label** -- If present, merge is blocked with an error
2. **Check for `release:*` label** -- Must have exactly one. If missing or multiple, CI fails
3. **Report the channel** -- The detected channel is logged for downstream workflows

### Decision Tree

```
PR opened/updated
  |
  +-- Has "Holding" label?
  |     YES -> BLOCK merge, post comment, CI fails
  |
  +-- Has no "release:*" label?
  |     YES -> FAIL CI: "Add a release channel label"
  |
  +-- Has multiple "release:*" labels?
  |     YES -> FAIL CI: "Use exactly one release label"
  |
  +-- Has "release:skip"?
  |     YES -> Allow merge, no publish
  |
  +-- Has "release:latest"?
  |     YES -> Merge to main, publish to npm latest + Docker latest + GitHub Release (stable)
  |
  +-- Has "release:experimental"?
  |     YES -> Merge to experimental branch, publish to npm experimental + Docker experimental + GitHub Pre-release
  |
  +-- Has "release:canary"?
        YES -> Merge to canary branch, publish to npm canary + Docker canary + GitHub Pre-release
```

### Publish Flow

#### Latest (Stable)

Triggered by pushing a version tag to `main`:

```bash
# After PR with release:latest is merged to main
git tag v1.2.1
git push origin v1.2.1
```

Result:
- `npm publish --tag latest`
- Docker: `latest` + `v1.2.1`
- GitHub Release (stable)
- MCP Registry publish

#### Experimental

Triggered by pushing a version tag to `experimental`:

```bash
# After PR with release:experimental is merged to experimental
git tag v1.2.1-experimental.1
git push origin v1.2.1-experimental.1
```

Result:
- `npm publish --tag experimental`
- Docker: `experimental` + version tag
- GitHub Pre-release

#### Canary

Automatic nightly build from `canary` branch at 02:00 UTC:

- Auto-versioned: `1.2.x-canary.202604170200.abc1234`
- `npm publish --tag canary`
- Docker: `canary`
- GitHub Pre-release

Can also be triggered manually via GitHub Actions.

## Examples

### Example 1: Stable feature release

```bash
# 1. Create PR
gh pr create --title "feat: add new feature" --label "feature" --label "release:latest"

# 2. After review and merge, tag and push
git tag v1.2.0 && git push origin v1.2.0
```

### Example 2: Security hotfix

```bash
gh pr create --title "fix: patch CVE-XXXX" --label "security" --label "hotfix" --label "release:latest"
```

### Example 3: Experimental feature (not ready for stable)

```bash
gh pr create --title "feat: new experimental API" --label "feature" --label "release:experimental"
```

### Example 4: Work in progress

```bash
gh pr create --title "feat: HSM support" --label "feature" --label "security" --label "Holding"
# CI will block merge. Remove "Holding" when ready.
```

### Example 5: CI/docs change (no publish needed)

```bash
gh pr create --title "docs: update README" --label "documentation" --label "release:skip"
```

### Example 6: Dependency bump

```bash
gh pr create --title "deps: bump hono to 4.12.14" --label "dependencies" --label "security" --label "release:latest"
```

## Branch Protection Rules

The `main` branch requires:

1. **`label-check` CI must pass** -- Ensures proper release label
2. **No `Holding` label** -- Blocks accidental merge of incomplete work
3. **All CI tests pass** -- 3 Node.js versions (20, 22, 24)
4. **CodeQL analysis passes** -- Security scanning
5. **Review approval** -- At least 1 approving review

## Workflow Files

| File | Purpose |
|------|---------|
| `.github/workflows/label-check.yml` | Validates PR labels, blocks merge on violations |
| `.github/workflows/publish.yml` | Publishes to npm/Docker/GitHub based on channel, generates changelog |
| `.github/workflows/canary.yml` | Nightly canary builds |
| `.github/workflows/stale.yml` | Auto-closes stale issues/PRs after 60 days |
| `.github/workflows/welcome.yml` | Welcome messages for first-time contributors |
| `.github/workflows/ci.yml` | Test suite (Node 18, 20, 22) |
| `.github/workflows/codeql.yml` | Security analysis |
